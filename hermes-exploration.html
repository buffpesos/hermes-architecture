What Hermes is

Hermes Agent (by Nous Research) is a personal AI agent with one defining idea: a single agent core that runs identically across many surfaces — a terminal CLI, a TUI, a messaging gateway (Telegram / Discord / Slack / ~20 platforms), an Electron desktop app, and an IDE integration (ACP).

Its headline feature is a closed learning loop: it writes skills from experience, improves them as it uses them, curates memory with periodic nudges, models who you are across sessions, and searches its own past conversations. It runs anywhere — a $5 VPS, a GPU cluster, or serverless backends that hibernate when idle.

It is extended primarily through plugins and skills, not by growing the core.

Two governing principles

Almost every design decision in the repo traces back to two rules stated in AGENTS.md. They're the lens for reviewing any change — and they explain the shapes of every diagram below.

1 · Per-conversation prompt caching is sacred.

A long conversation reuses a cached prefix every turn. Anything that mutates past context or rebuilds the system prompt mid-conversation invalidates that cache and multiplies the user's cost — so it is forbidden. The one exception is context compression.

2 · The core is a "narrow waist"; capability lives at the edges.

Every tool ships on every API call, so adding a core tool is expensive. New capability is pushed outward along a Footprint Ladder: extend existing code → CLI command + skill → service-gated tool → plugin → MCP server → (last resort) a new core tool.

Prompt caching — the "why" behind the design

Principle #1 ("caching is sacred") drives a surprising number of design decisions, so it's worth making concrete. A system prompt is Hermes's own hidden instructions to the model — "you are Hermes, here are your tools, here's the user's context" — sent before your message on every turn. It's the same for the whole conversation, and you never see it.

First turn builds; later turns restore

These are two different events that are easy to conflate. On the first turn there is nothing saved, so Hermes builds the system prompt and saves it to the session row. On every later turn it reads that saved copy back instead of rebuilding — so the bytes are identical. (There is nothing to "restore" on turn 1; turn 1 is what creates the saved copy.)

Why only the prefix, and how much?

An LLM reads the prompt left → right, and the work it does for any token depends on every token before it. So the provider can reuse saved work for token #500 only if tokens #1–#499 are byte-identical to last time. The instant one early byte differs, everything after it must be recomputed. That's why caching can only be a prefix — an unbroken run from the very start. "How much" is cached = the longest run, from the start, identical to the previous request (up to the provider's limit). Because each turn appends to the end, a bigger share of the prompt is cached as the conversation grows — which is what keeps long conversations cheap.

Provider does the caching; the client earns the hit

Caching is a provider-side feature, and most big providers offer it — but not identically. OpenAI caches prefixes automatically. Anthropic does not — you must explicitly mark cache_control breakpoints, or you get zero caching (Hermes does this via apply_anthropic_cache_control, conversation_loop.py:817, marking system + last 3 messages). Other endpoints vary. The deeper point: the provider can only reuse the cache if the incoming prefix matches — and the provider has no control over what bytes you send. Only the client can keep the prefix stable. If Hermes rebuilt the prompt slightly differently each turn, the provider's cache would sit there and miss every time.

This is why so many decisions elsewhere in the doc read "injected into the user message, not the system prompt": memory context, skill bodies, and plugin context all go at the end precisely so the prefix never changes. Context compression is the one sanctioned exception — it rewrites history on purpose, and is therefore done deliberately and rarely.

Architecture overview

The mental model: many surfaces on top, exactly one core, and a wide fan of capabilities and external services at the edges. The core (run_agent.py) talks to a session store and to LLM providers, exposes a tool layer, and everything else plugs in around it.

The agent loop — one turn, start to finish

AIAgent.run_conversation() (run_agent.py:5302) is a thin forwarder; the real machine is agent/conversation_loop.py (~4,600 lines). One "turn" (one user message → one final reply) has three phases: pre-turn setup → the tool-calling loop → finalize.

Reading the diagram against the code

  • The while condition (conversation_loop.py:589): while (api_call_count < max_iterations and iteration_budget.remaining > 0) or _budget_grace_call. Two independent limits — a hard iteration cap (default 90) and an IterationBudget — plus a one-turn grace call so an exhausted budget still lets the model wrap up cleanly.
  • Interrupt-first. Every iteration checks _interrupt_requested (:594) and breaks — this is what lets a new Telegram message or Ctrl+C stop the agent immediately. A /steer drain (:662) injects mid-flight guidance by appending to the last tool-result message (never a new user message, which would break role alternation).
  • Cache control is applied at request-build time, not stored. The system prompt is restored from the prior turn's cached copy (_restore_or_build_system_prompt, :255) so the Anthropic prefix matches byte-for-byte. The file is full of log lines whose only job is to make a silent cache miss visible. Principle #1 as real code.
  • The branch (:3812): if assistant_message.tool_calls → _execute_tool_calls(...) (:4079), append results, loop again. Otherwise final_response = assistant_message.content (:4180) and break.
  • Tool execution can be concurrent. _execute_tool_calls (run_agent.py:5188) sends a safe multi-tool batch to execute_tool_calls_concurrent (agent/tool_executor.py), else runs sequentially. A guardrail can cleanly halt the turn instead of crashing.
  • Finalize (:4576 → agent/turn_finalizer.finalize_turn) is where the learning loop hooks in: persist the session, run memory sync_all, and fire the skill nudge if enough tool-iterations passed without using skill_manage (_iters_since_skill, incremented at :648).

Takeaway: most of the loop is defensive — empty-response retries, message-alternation repair, compression-on-overflow, truncated tool-call recovery. The machinery exists to keep a long-lived, cache-sensitive conversation from wedging.

Tools & toolsets — the narrow waist in practice

When the loop hits a tool call it lands in handle_function_call() (model_tools.py:901). Three files cooperate: tools/registry.py is the system of record, toolsets.py decides which tools are visible, and model_tools.py orchestrates discovery and dispatch.

Build-time — discovery & gating

Every file in tools/ calls registry.register(name, toolset, schema, handler, check_fn=…) at import (tools/registry.py:234). discover_builtin_tools() AST-scans the directory, so nothing is manually listed. Then get_tool_definitions() resolves the session's enabled_toolsets, subtracts disabled_toolsets, and runs each tool's check_fn (30-second cached). A tool whose gate returns false is simply absent from the schema list — that's how terminal only appears when a backend is reachable, Home-Assistant tools only when a token is set, and so on.

Why this is the narrow waist: the resolved schemas are sent on every API call, so the schema list is the cost surface. _HERMES_CORE_TOOLS (toolsets.py:31) is the always-on set (terminal, file, web, browser, vision, skills…). Everything else is gated, and when the deferred surface grows it's hidden behind a tool_search / tool_describe / tool_call bridge so it doesn't bloat the prompt.

Run-time — one tool call, end to end

A tool call passes, in order, through: the tool-search bridge (unwraps tool_call to the real name so hooks see it) → request middleware (plugins may rewrite args) → the pre-tool-call hook + approval gate (a plugin or a user approval can block it — dangerous-command confirmation lives here) → registry.dispatch() runs the handler. Async handlers are bridged onto a persistent event loop (_run_async) to dodge "event loop is closed" with cached httpx clients; exceptions are sanitized into {"error": …} so a crash can't poison the conversation; post-hooks observe; a JSON string returns to the loop.

Terminal backends

Everything under tools/environments/ shares a BaseEnvironment ABC with a spawn-per-command model. TERMINAL_ENV config selects the substrate — local (default), docker, ssh, modal / managed_modal, singularity, or daytona. Same terminal tool, swappable execution backend — which is what lets the agent "run anywhere," including serverless backends that hibernate when idle.

The learning loop — what makes Hermes "self-improving"

This is the headline claim, and it's a genuine closed loop across four subsystems. The insight that ties them together: everything learned is re-injected as user messages (never the system prompt), so the loop never breaks prompt caching.

Skills

A skill is a SKILL.md with YAML frontmatter, living in skills/ (built-in), ~/.hermes/skills/ (user / agent-created), or optional-skills/ (shipped but off by default). scan_skill_commands() (agent/skill_commands.py:348) turns each into a /slash-command. The cache-critical detail: invoking a skill never touches the system prompt — build_skill_invocation_message() (:517) loads the whole body into a user message behind an [IMPORTANT: the user invoked '{skill}'…] marker. Creation is via /learn (build_learn_prompt(), learn_prompt.py:72) — the agent gathers sources with its normal tools and authors one SKILL.md via skill_manage action=create — or autonomously: the _iters_since_skill counter from the agent loop fires a nudge after a complex multi-tool task, prompting the agent to persist what it just figured out.

Curator — skills don't accumulate forever

"Walks a lifecycle state machine" is literal: each skill carries an explicit stored state, and a function flips it based on inactivity.

  • Where the state lives: an explicit state field ("active"/"stale"/"archived") in ~/.hermes/skills/.usage.json — a plain JSON sidecar file keyed by skill name (tools/skill_usage.py), not the SQLite database (that stores conversations). The skill library is files on disk, so its metadata sits beside it. The record also holds use_count, last_used_at, pinned; the curator's own run metadata (last_run_at, run_count, paused) lives in a second JSON file, ~/.hermes/skills/.curator_state. "Archived" is both a state value and a physical move into .archive/; "stale" is status-only.
  • Who moves them: apply_automatic_transitions() (curator.py:276) computes stale_cutoff = now − 30d and archive_cutoff = now − 90d, then per skill: idle>30d & active → stale; idle>90d → archive; reused-after-stale → active; pinned → skipped.
  • Who triggers it: maybe_run_curator(), called at CLI startup (cli.py) and on the ~hourly gateway housekeeping tick (gateway/run.py), gated by should_run_now() (curator.py:219) to run at most once every 7 days. Scope: agent-created always; bundled built-ins only if curator.prune_builtins; hub-installed and protected built-ins (e.g. plan) never.

Optionally (off by default) the curator also spawns an isolated agent fork to consolidate overlapping skills into umbrellas. Every mutation is preceded by a tar.gz snapshot (agent/curator_backup.py) — which also backs up cron/jobs.json, since consolidation can rewrite skill references jobs point at.

Memory & user modeling

MemoryManager (agent/memory_manager.py) orchestrates the built-in memory plus at most one external provider. Two hooks bracket every turn: prefetch_all(query) recalls relevant context before the model call; sync_all(user, assistant) persists after. The flagship external provider is Honcho (plugins/memory/honcho) — "dialectic" user modeling that injects an evolving user representation + peer card and, on a configurable cadence, runs multi-pass reasoning about you. All injected into the user message in <memory-context> fences, with a StreamingContextScrubber (:132) stripping any leaked fences from the model's output.

Cross-session recall

This is how the agent reaches into past conversations (distinct from memory and from resuming a session).

  1. The agent calls session_search (tools/session_search_tool.py) — a normal gated tool, invoked when you reference earlier work. Not automatic.
  2. It runs FTS5 over all sessions. SessionDB.search_messages() (hermes_state.py:3466) joins messages_fts → messages → sessions with no session-id filter by default, ranked by BM25. The missing filter is exactly what makes it cross-session.
  3. It returns raw, structured hits — no LLM. The schema says so verbatim: "No LLM calls, no summarization." Per match it returns a bookend + window shape: the session's opening messages (the goal), a ±5-message window around the match (the detail), and the closing messages (the outcome) — enough to reconstruct goal → match → resolution without the full transcript.
  4. The agent's own model does the "summarizing." Hits come back as a tool-result message; the model reading them turns snippets into "three weeks ago we set the Docker backend to X." So the README's "FTS5 search with LLM summarization" is accurate, but the LLM step is the agent itself, not code inside the tool.

This is a different system from memory. Memory (MEMORY.md, Honcho) = curated facts surfaced automatically in context; recall = on-demand full-text search over raw transcripts. They share no code path. Both also differ from resuming a session, which loads one old conversation's full history via get_messages() to continue it.

Why FTS5?

FTS5 is a full-text search engine built into SQLite — virtual tables that index text and answer ranked keyword/phrase/boolean/prefix queries. Hermes keeps two: messages_fts (unicode61) and messages_fts_trigram (for CJK substring search), both kept in sync by triggers on every message insert/update/delete that index content + tool_name + tool_calls together. It fits Hermes's constraints exactly:

  • Zero dependencies, fully local. The store is already SQLite; FTS5 ships inside it. No Elasticsearch, no embedding service, no network — runs on a $5 VPS.
  • Free to maintain. Trigger-driven, so the index can't drift from the transcript; writing a message is indexing it (no separate job).
  • Ranking + snippets built in. BM25 for "most relevant first", snippet() for >>>highlighted<<< previews — exactly what you hand a model.
  • CJK robustness. The trigram table + a LIKE fallback for 1–2 character queries handle scripts the default tokenizer mangles.
  • Keyword, not semantic — on purpose. Cheap, deterministic, explainable; "semantic" recall is delegated to the model reading the results and to the separate memory/Honcho layer.

/insights (agent/insights.py:84) reuses the same store to aggregate historical sessions into token / cost / tool / skill / platform breakdowns.

Cron — autonomous, scheduled work

Cron is a clean illustration of the Footprint Ladder: the what-to-run core is built-in (cron/jobs.py, cron/scheduler.py), the when-to-fire is a pluggable provider, and the user surface is both an agent tool and a CLI command — no new core model-tool was needed for the scheduling logic.

The job model

cron/jobs.py:724 defines a JSON record in ~/.hermes/cron/jobs.json: a prompt, a schedule (parsed into once / interval / cron-expression kinds), repeat count, deliver target, optional skills, context_from (output of other jobs piped in — chainable pipelines), workdir, per-job model/provider, and no_agent. Mutations take a cross-process file lock so multiple gateway replicas on one HERMES_HOME don't corrupt the file.

Execution

The tick (tick(), scheduler.py:2384) grabs a non-blocking lock, finds due jobs, and advances next_run_at before dispatch — that's how recurring jobs get at-most-once semantics even across a crash. Then run_job() (:1597) takes one of two paths: a no_agent script (plain subprocess, with a {"wakeAgent": false} convention to stay silent), or a full AIAgent spun up with platform="cron", a fresh session id, the job's skills loaded, upstream outputs injected, and a "you are running as a scheduled cron job" hint. It even scans the assembled prompt for injection threats before running, since cron prompts run unattended.

Delivery & the Footprint Ladder

Delivery (scheduler.py:514) is decoupled from execution: output is saved to disk first (so it survives a delivery failure), then routed to the target — telegram, discord, local, origin (reply where the job was created), or comma-separated fan-out — preferring the live gateway adapter and falling back to a standalone send. last_error and last_delivery_error are tracked separately.

Where it sits on the ladder: the agent schedules its own jobs through the cronjob() tool (tools/cronjob_tools.py:522); you manage them with hermes cron …; and the trigger is a CronScheduler ABC. The default is a 60-second in-process ticker, but a plugin like Chronos (docs/chronos-managed-cron-contract.md) replaces it with externally-armed one-shots, so the agent process can scale to zero between fires and wake on demand — the "costs nearly nothing when idle" claim made concrete.

Session storage — the SessionDB

All conversation state lives in one SQLite database (hermes_state.py, the SessionDB class), profile-aware under ~/.hermes/. It is both the transcript store and the search index, and it's the layer that makes prompt-cache restore possible across turns.

The schema (the parts that matter)

TableWhat it holds
sessionsOne row per conversation: id, source (cli/telegram/…), model, system_prompt (the cached prefix, restored each turn), parent_session_id (subagent / handoff lineage), token + cost counters, title, archived, rewind_count.
messagesOne row per message: role, content, tool_calls, tool_name, reasoning*, timestamp, token_count, and three state flags — active (0 after a rewind), compacted (1 if folded into a compression summary), observed.
messages_fts + messages_fts_trigramFTS5 virtual tables for full-text and CJK-substring search, kept in sync by triggers on every message insert/update/delete.
state_meta, compression_locksKey/value state and advisory locks coordinating compression across processes.

How writes happen

Messages are flushed by _flush_messages_to_session_db() (run_agent.py:1581) — called from multiple exit points (early crash-safe persist, after each tool batch, at finalize). It tracks message object identity in a _flushed_db_message_ids set so repeated calls only write genuinely new rows (fixing a duplicate-write class of bug), and it survives message-sequence repair that would break a positional cursor. Multimodal tool results are reduced to a text summary before storage so base64 images never bloat the DB.

Why this design

  • Search is free. The FTS index is trigger-maintained, so the agent searching its own past (search_messages(), :3466) needs no separate indexing pass — every message is searchable the instant it's written.
  • Cache restore is a DB read. Storing the system_prompt per session is what lets a fresh AIAgent (every gateway turn builds one) restore the byte-identical prefix and keep the Anthropic cache warm.
  • Rewind/compress without loss. active/compacted flags mean /undo and compression hide rows rather than delete them — history stays intact and searchable.
  • Lineage & cost are first-class. parent_session_id threads subagents and cross-platform handoffs; per-session token/cost columns feed /usage and /insights.

Request trace — sessions × the learning loop

To see both systems interact, follow one CLI request — say you type "refactor this module the way I like it." The diagram is a sequence across three actors (the turn loop, the SessionDB, and the memory/skills subsystem); the text below maps each arrow to code.

Pre-turn — agent/turn_context.py

  1. Session row created — _ensure_db_session() (run_agent.py:524) inserts the sessions row (system_prompt initially NULL).
  2. Cached prompt restored, not rebuilt — restore_or_build_system_prompt (conversation_loop.py:254) reads sessions.system_prompt and restores it verbatim if it matches the runtime, so the prefix cache hits; a fresh session builds once and writes it back via update_system_prompt (:368).
  3. Your message persisted before the model call — _persist_session → _flush_messages_to_session_db (run_agent.py:1581). Crash-safe by construction.
  4. Memory prefetched — MemoryManager.prefetch_all(query) (memory_manager.py:453) asks every provider; the result is wrapped by build_memory_context_block and injected into the user message (conversation_loop.py:750), never the system prompt.
  5. Counters tick — _turns_since_memory (may set should_review_memory) and, inside the loop, _iters_since_skill. These seed the post-turn self-improvement.

Loop

  1. The model is called against the cached prefix. As tools run, new assistant/tool messages are appended incrementally (identity-tracked, idempotent). Every insert fires the FTS triggers — your conversation is searchable as it's written, no separate index step. active/compacted flags keep rewinds and compression non-destructive.

Finalize — agent/turn_finalizer.py

  1. Final flush + stats — remaining messages written, token/cost counters updated.
  2. sync_all mirrors the turn into memory (turn_finalizer.py:417 → memory_manager.py:516) on a background executor, so a slow provider can't stall the turn.
  3. The self-improving fork — the key mechanism. If _iters_since_skill ≥ _skill_nudge_interval (default 10) or should_review_memory is set, _spawn_background_review (run_agent.py:1439) launches a separate AIAgent on a daemon thread, after your reply is already delivered. It reuses the parent's warm prefix cache, runs quietly with nested nudges disabled, and is handed the transcript plus a review prompt (agent/background_review.py): the skill prompt says "be active — if the user corrected your style or workflow, embed it in SKILL.md now"; the memory prompt asks it to save persona/preferences. Its only meaningful tools are skill_manage and memory, so its output is a write to your skill library / MEMORY.md.
The punchline.

"Creates skills from experience" and "nudges itself to persist knowledge" are implemented as a shadow agent that reflects on each conversation and writes it down — invisibly, off the critical path, and without touching the user-facing conversation's prompt cache. Self-improvement and session storage are the same story told twice: the SessionDB is the durable record, and the background fork is what turns that record into lasting behavior.
